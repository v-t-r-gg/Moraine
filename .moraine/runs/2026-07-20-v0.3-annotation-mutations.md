# Operation-based annotation mutations

## Objective

Eliminate whole-list annotation replacement and prevent concurrent
annotation changes from being silently lost. Harden suggestion accept/reject
so outcomes are durable and acceptance cannot leave Markdown changed without
a recoverable ledger state.

## Related issue

GitHub issue #2. PR #3.

## Scope

- Operation-based annotation mutations under lock.
- Per-annotation revision concurrency tokens (checked increment).
- Durable suggestion disposition (pending / accepting / accepted / rejected / resolved_legacy).
- Two-phase suggestion acceptance bound to Markdown content hash.
- Revision-safe cancellation: cancel only when disk hash equals acceptance base hash.
- Incomplete acceptance recovery (cancel when safe; explicit finalize when document changed).
- Session reconcile without full-list replace or deletes.

## Out of scope

- Richer annotation anchors, reattachment UI, evidence, inboxes, auth.

## Durable suggestion state model

| Disposition | Meaning |
| ----------- | ------- |
| pending | Open suggestion |
| accepting | Reservation after begin_accept; incomplete until complete/cancel |
| accepted | Applied and finalized after successful Save |
| rejected | Explicit reject |
| resolved_legacy | Pre-v3 resolved suggestion; outcome unknown |

## Two-phase acceptance and recovery

1. `begin_accept_suggestion` (annotation rev + Markdown hash) → `accepting`
2. Apply in editor + host Save
3. `complete_accept_suggestion` (op id + saved hash) → `accepted`
4. `cancel_accept_suggestion` only if **current disk hash == acceptance base hash**
5. If disk hash differs: cancel fails with `acceptance_document_changed`; human may **Finalize** for current saved revision or restore original first

## Selected concurrency-token design

Per-annotation `revision` with checked_add. Overflow → `revision_overflow`.
New reconcile IDs always force revision `1`.

## Schema and compatibility effects

- Schema version **3**.
- Legacy resolved suggestions → `resolved_legacy`.
- Decisions and run IDs preserved.

## Live-session persistence behavior

- Host ops for local mutations; structured JSON error kinds in IPC error strings.
- Host Save reconcile: no deletes; protect accepting/terminal outcomes.
- Browser-only: session only until host persistence.

## Manual defect discovered

During idle browsing and review (no intentional external edit), the UI flashed:

`The file changed on disk. Reload or resolve a conflict before recording a decision.`

Effects: unreadable flash, scroll jumps to top, autosave/reload loop. Approvals still persisted, but the editor was unusable.

## Confirmed root cause

1. Host Save writes Markdown via atomic replace (multiple FS events).
2. Soft suppress only counted once; extra events reached the UI.
3. Frontend set `externalConflict = true` on every notification before comparing hashes.
4. Clean docs auto-reloaded with `applyDocument(..., resetSession: true)`, destroying the Yjs session and scroll position.
5. Programmatic setContent could re-dirty and schedule autosave, repeating the cycle.

## Watcher revision-classification design

- Save response includes authoritative `contentHash` (SHA-256 of persisted UTF-8).
- Backend updates `knownContentHash` immediately on open/save/reload.
- Watcher emits only when disk hash differs from known and is not a duplicate external hash.
- Sidecar paths are ignored for Markdown conflict handling.
- Frontend classifies via `classifyDiskEvent` (equal hash / duplicate / saving / clean / dirty).

## Viewport-preservation behavior

Same-file external reload uses `applyDocumentInPlace`: keeps Yjs session, sets Markdown programmatically, restores scroll/selection via `getViewportState` / `restoreViewportState`.
