# Operation-based annotation mutations

## Objective

Eliminate whole-list annotation replacement and prevent concurrent
annotation changes from being silently lost. Harden suggestion accept/reject
so outcomes are durable and acceptance cannot leave Markdown changed without
a recoverable ledger state.

## Related issue

GitHub issue #2. PR #3.

## Scope

- Operation-based annotation mutations under lock.
- Per-annotation revision concurrency tokens (checked increment).
- Durable suggestion disposition (pending / accepting / accepted / rejected / resolved_legacy).
- Two-phase suggestion acceptance bound to Markdown content hash.
- Session reconcile without full-list replace or deletes.

## Out of scope

- Richer annotation anchors, reattachment UI, evidence, inboxes, auth.

## Starting point

- Branch: `milestone/v0.3-durable-annotations`
- Base: `1bd5d23` / tag `review-ledger-v0.2.1`

## Durable suggestion state model

| Disposition | Meaning |
| ----------- | ------- |
| pending | Open suggestion |
| accepting | Reservation after begin_accept; incomplete until complete/cancel |
| accepted | Applied and finalized after successful Save |
| rejected | Explicit reject |
| resolved_legacy | Pre-v3 resolved suggestion; outcome unknown |

Comments still use open/resolved only.

## Two-phase acceptance

1. `begin_accept_suggestion(id, expectedRevision, expectedMarkdownHash)` → disposition accepting, op id, rev++
2. Editor apply + host Save
3. `complete_accept_suggestion(id, expectedRevision, opId, savedHash)` → accepted
4. On apply/Save failure: `cancel_accept_suggestion` → pending

## Selected concurrency-token design

Per-annotation `revision` with checked_add. Overflow → `revision_overflow`.
New reconcile IDs always force revision `1` (ignore client revision).

## Schema and compatibility effects

- Schema version **3**.
- v2 loads with disposition normalized in memory; writes promote to v3.
- Legacy resolved suggestions → `resolved_legacy` (not guessed accept/reject).
- Decisions and run IDs preserved.

## Live-session persistence behavior

- Host ops for local mutations; structured JSON errors to UI.
- Host Save: reconcile by ID; never delete; protect accepting and terminal outcomes from stale pending snapshots.
- Browser-only: session only until host persistence.

## Tests and evidence

```bash
./scripts/check.sh   # exit 0 after hardening
```

Includes two-phase accept, concurrent begin, reject/accept opposite failures, reconcile vs accepted/accepting, revision-1 force on new session ids.

## Known limitations

- Accept requires a clean saved base hash before begin.
- If finalize fails after a successful Save of applied text, disposition stays accepting until cancel/complete.
- No multi-file ACID transaction across Markdown and ledger.
