# Operation-based annotation mutations

## Objective

Eliminate whole-list annotation replacement and prevent concurrent
annotation changes from being silently lost.

## Related issue

GitHub issue #2.

## Scope

- Inventory all annotation mutation paths.
- Replace full-list writes with operation-based mutations.
- Re-read the latest ledger under the per-document lock.
- Preserve concurrent operations on different annotations.
- Return explicit conflicts for incompatible operations on the same annotation.

## Out of scope

- Richer annotation target selectors.
- Quote reanchoring.
- Manual reattachment UI.
- Evidence records.
- Review inboxes.
- Collaboration hardening.

## Starting point

- Branch: `milestone/v0.3-durable-annotations`
- Base commit: `1bd5d23`
- Related release tag: `review-ledger-v0.2.1`

## Invariants (from issue #2)

- Concurrent operations on different annotations must both survive.
- Incompatible operations on the same annotation must not silently use last-writer-wins and must return an explicit conflict or failed precondition.

## Selected concurrency-token design

**Per-annotation monotonic `revision` (u32).**

- Missing field → default `1` (schema v2 compatible).
- Create starts at `1`.
- Successful mutation → `revision + 1`.
- Callers pass `expectedRevision`; mismatch → `annotation_conflict`.

Rationale: simpler than fingerprints, stable serde defaults, easy UI wiring.

## Operation semantics

| Op | Precondition | Effect |
| -- | ------------ | ------ |
| create | unique ID; comment body non-empty | Append rev=1 |
| update | exists; expected rev | body/author; rev++ |
| resolve | exists; expected rev | resolved=true; idempotent if already |
| reopen | exists; expected rev | resolved=false; idempotent if already |
| accept_suggestion | kind=suggestion; expected rev | resolved=true (doc edit stays in UI) |
| reject_suggestion | kind=suggestion; expected rev | resolved=true |

## Live-session persistence behavior

- Browser peers write Yjs only.
- Host mutations call Tauri ops with expected revision; Yjs updated from durable result.
- Host Save calls `reconcile_session_annotations`: create missing IDs, update when revision matches, never delete disk-only, conflict keeps disk.
- Remote peer annotations become durable when the host saves (reconcile) or when the host performs ops that already include them in Yjs before save.

## Schema and compatibility effects

- Additive `revision` on comments; schema version remains **2**.
- Decisions and run IDs unchanged.
- No production full-list annotation write path.

## Known limitations

- Accept applies document text in the editor first; if the ledger accept then conflicts, the document may already have changed (reported clearly).
- No public annotation CLI (concurrency covered by core process tests).
- Live multi-host desktop still relies on revision tokens; reconcile on Save is best-effort for remote Yjs-only creates.
- Reviewer labels remain unauthenticated.
